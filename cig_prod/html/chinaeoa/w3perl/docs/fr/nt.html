Serveurs NT
Historique W3perl a été complètement testé sur un serveur NT 4.0 grace à Alexandre Gefen.
La première tentative pour faire tourner w3perl sur un serveur NT a été faite avec succès par J-L Damez mais demandait alors de nombreuses arrangements manuels. Tout a été simplifié depuis car il vous suffit juste de sélectionner l'option NT dans le fichier de configuration.
Vous devrez installer ActivePerl et Fly.exe pour faire tourner w3perl.
Vous pouvez utiliser la commande at pour lancer les scripts. Cron-w3perl.pl ne peut pas être utilisé car un serveur NT ne peut pas forker des processes comme sous Unix.
Cron-pages sert juste une fois pour l'initialisation des stats. Chaque jour, lancez cron-inc, cron-day et cron-hour. Chaque semaine, lancez cron-week, cron-session. Finalement chaque mois, lancez cron-month.

Installation La procédure d'installation a été grandement simplifiée sous NT depuis que j'ai pu faire par moi même cette installation. Il semble qu'il n'est pas possible d'utiliser les variables d'environnement sur un serveur Microsoft :( donc vous devrez éditer fixperlpath.pl et modifiez la deuxième ligne en accord avec le chemin où se trouve w3perl sur votre machine. Il s'agit de la seule et unique spécificité que vous ayez à faire pour l'installation, plus de détails sont disponibles dans la section d' installation.

Limitations Les serveurs NT peuvent produire des fichiers de log au format CLF (Common Logfile Format) ou au format IIS format. W3Perl gére ces deux formats, il est juste nécessaire de sélectionner lequel vous utilisez dans le fichier de configuration.

IIS 3.0 n'a pas de support pour le format extended CLF, par conséquent les informations sur les fureteurs et les références ne sont pas accessibles dans la distribution standard. Mais une personne m'a avertit de la possibilité d'avoir ces informations dans le fichier de log de IIS 3.0 en téléchargeant une dll fournie par microsoft.
W3Perl peut aussi bien utiliser des fichiers de log issus de serveurs IIS 4.0 que de serveurs IIS 3.0 (avec ou sans modification) pour effectuer des stats sur les fureteurs et les références.

Les erreurs sur un serveur NT sont stockées dans un gestionnaire d'événements et par conséquent aucun fichier n'est produit dans ce cas. Mais vous pouvez utiliser une redirection sur des pages HTML pour les différents codes d'erreurs possibles, ainsi vous aurez les erreurs stockées dans votre fichier de log.

Nouveautés Le support pour des fichiers utilisant la compression zip, le support pour les fichiers de log compressés quotidiennement a été effectué (les fichiers de log journaliers sont l'option par défaut sur un serveur IIS).
